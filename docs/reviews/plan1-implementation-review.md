`SchemaV2.swift` is close to `docs/data-schema.md`, and I did not find direct SQL injection issues: the stores consistently use prepared statements, and the one dynamic `IN (...)` clause only interpolates placeholder tokens, not user data. The bigger issue is that the implementation is only partially migrated to the new schema. `DataManager.swift` still ships legacy `progress_snapshot` APIs and older `logReview` / `startSession` methods that do not match the V2 tables, and `initializeIfNeeded()` does not create the required Day 0 `day_state` row from the schema lifecycle.

The most important logic bug is in `WordStateStore.recordScoredAnswer()`: it promotes box-0 words on any correct answer and always increments `day_touches`, which conflicts with the spec’s Day 1 flow where new words stay at box 0 until the end-of-evening promotion decision and rushed answers should not count toward promotion. `runDay1Promotion()` has a second correctness issue: it does not filter its `review_log` counts by `study_day`, so historical attempts can affect today’s promotion result.

Actor isolation is also shaky. `DataManager` exposes a single `nonisolated` SQLite connection, while `StatsStore`, `SessionStateStore`, and `ZoneStore` are separate actors that all use that same handle, so actor boundaries do not actually serialize DB access. Finally, several update-only methods can fail silently at the data level: `StatsStore` and `SessionStateStore` often `UPDATE` rows without first ensuring they exist or checking affected-row count, and `parseSessionState()` swallows bad JSON with `try?`, which can erase paused-session state instead of surfacing corruption.
